1. Problem Statement
An internal IT support team handles hundreds of tickets per week arriving via email, Slack, and a help portal. Today, every ticket requires a human to read and manually route it to one of 7 resolver queues before any resolution begins. This creates triage bottlenecks, delays, and wastes skilled engineer time on a task a machine can learn.
TicketRouter-1.7B replaces that first human touch. It reads the ticket, predicts the correct resolver queue, and routes it automatically. The model never writes a customer response — it only classifies. The downstream support tooling takes over from there.
The core question this project answers: can a 1.7B parameter model, fine-tuned on labeled examples, achieve reliable routing accuracy at a fraction of frontier model cost?
2. Architecture Decision — Why LoRA Fine-Tuning
Before training anything, the right question is: is this a knowledge problem or a behavior problem? Routing is behavior — the model needs consistent classification patterns, not retrieved facts. That rules out RAG immediately.
Why LoRA specifically: LoRA trains two small matrices (A and B) while base model weights stay frozen. Formula: W = W₀ + AB. This gives ~10,000x fewer trainable parameters, 3x less GPU memory than full fine-tuning, and makes a free Colab T4 sufficient. The adapter encodes routing behavior without touching the base model.

8. Final Results — All 4 Runs vs Base Model
Run 2 is the production candidate. +53.0% accuracy over base. 5 of 7 classes went from zero baseline capability to strong performance. Deployed to HuggingFace as Neog007/TicketRouter-1.7B.
9. Key Learnings from 4 Runs
▸  Targeted augmentation beats random volume. 40 domain-specific examples (Run 2) outperformed 95 mixed examples (Run 3 + Run 4).
▸  Quality and signal specificity matter more than quantity — proven empirically across 4 runs.More data is not always better.
▸  Run 3 changed data AND epochs simultaneously. Run 4 isolated epochs and confirmed data quality was the culprit.Isolate one variable at a time.
▸  The same 117 examples evaluated all 4 runs — making every comparison fair and meaningful.Hold validation constant.
▸  Fine-tuning didn't just improve performance — it created performance that didn't exist.The base model had zero routing capability on 5 of 7 classes.
▸  AD examples used 'security group', 'domain account', 'MFA enrollment' — real IT terminology that taught the model what AD tickets sound like.Domain language drives augmentation quality.
10. Cost Analysis — Why Fine-Tuning Wins at Scale
8-Stage Cost Breakdown
Break-Even vs API Routing
Privacy & compliance: IT support tickets contain usernames, account names, device IDs, internal system names, and access requests. Local inference keeps all of that inside the organization boundary. No ticket data is sent to a third-party API. For regulated industries this is a compliance requirement, not a preference.

Streamlit Live Demo
A production-ready Streamlit app is deployed publicly. Users paste any IT support ticket, click Route Ticket, and receive the predicted queue with confidence score in real-time. Demonstrates the model works end-to-end outside a notebook environment.
Note on inference backend: The live demo uses the Claude API as the inference backend rather than loading the fine-tuned model directly. The merged TicketRouter-1.7B model (~3.4GB) exceeds Streamlit Cloud's memory limits for direct loading. The fine-tuned model is fully available at Neog007/TicketRouter-1.7B on HuggingFace for local inference. Production deployment would run on a dedicated GPU endpoint (e.g. HuggingFace Inference Endpoints, Replicate, or a self-hosted server) where the full model loads without memory constraints.
